作成日: 2026-02-16
目的: 「楽しい一日の再現性を持たせる」ための基盤づくり

## 概要

このスプリントでは、以下の5機能を実装する。詳細な設計・requirements.md更新・実装計画書は各機能の着手時に行う。

## 機能1: 作業時間計測

### 目的

日々の作業時間と達成レベル・満足度の相関を分析するための基礎データを収集する。

### 仕様概要

**計測開始**
- アプリ（ブラウザタブ）を開いた時点でタイマーをスタート

**計測継続/停止**
- タブがバックグラウンドになった時（`visibilitychange`イベント）: 一時停止
- タブが再度アクティブになった時: 再開
- タブを閉じた時（`beforeunload`イベント）: 停止 → 累積時間をlocalStorageに保存

**計測確定**
- 日報を保存した時点で、その日の累積時間を `daily_records` に保存して計測終了

**注意点**
- localStorageでセッション間の累積時間を引き継ぐ
- 当日の日報保存後に再びアプリを開いても、その日の計測は追加しない（確定済みのため）
- ブラウザクラッシュ等でlocalStorageに未保存の時間が残った場合は破棄（精度より堅牢性優先）

### データモデル変更（予定）

`daily_records` テーブルに以下を追加:

| カラム名 | 型 | NULL | 説明 |
|---------|-----|------|------|
| `work_duration_minutes` | INTEGER | 許容 | 作業時間（分）。日報保存時に確定 |

## 機能2: タイムラインページ（マイルーティン）

### 目的

「自分の理想の一日の流れ」をtodoリスト形式で育てていくページ。日付に依存せず蓄積し、毎日参照することで行動の改善を積み重ねる。

### 仕様概要

**ページの性質**
- 日付に紐づかない（マイルーティンとして永続的に育てていく）
- 既存の「その他todo」「Gold/Silver/Bronze目標todo」とは独立した管理

**表示形式**
- 時間帯タグ付きのリスト形式（例: `7:00 散歩に行く`、`9:00 開発作業`）
- 同一時間帯に複数のtodoを設定可能
- 時間帯順にソート

**todo操作**
- 追加: 時間帯（HH:MM形式）とtodo内容を入力して追加
- 削除（非表示）:
  - 削除時に「削除理由」の入力を必須とする
  - 削除されたtodoは物理削除せず、非表示フラグで管理（ソフトデリート）
  - 削除理由の選択肢（案）: 「満足度向上のため見直し」「今の状況でできない」「別のことを優先」「その他（自由記述）」
- 復活: 削除済みtodoを一覧から選択して復活させる機能

**日報との連携**
- タイムラインのtodoは日報のtodoとは独立（日報のtodoに自動反映はしない）
- ただし将来的に「タイムラインtodoをその日の日報に取り込む」機能を追加できる設計にしておく

### データモデル変更（予定）

新テーブル `timeline_todos` を追加:

| カラム名 | 型 | NULL | 説明 |
|---------|-----|------|------|
| `id` | UUID | - | 主キー |
| `user_id` | UUID | - | 外部キー: auth.users |
| `time_tag` | TEXT | - | 時間帯（例: "07:00"） |
| `content` | TEXT | - | todo内容 |
| `is_deleted` | BOOLEAN | - | ソフトデリートフラグ（default: false） |
| `delete_reason` | TEXT | 許容 | 削除理由（削除時に記録） |
| `created_at` | TIMESTAMPTZ | - | 作成日時 |
| `updated_at` | TIMESTAMPTZ | - | 更新日時 |

**アクセス**
- サイドバーのメニューに「ルーティン」として追加
- メニュー順序（案）: ホーム → 記録 → カレンダー → ルーティン → 目標変遷 → 設定

## 機能3: 週次振り返りページ

### 目的

週単位でGold/Silver/Bronzeの達成状況を振り返り、目標の調整判断に使う。Gold目標は週次振り返りページ経由でいつでも編集可能。Silver/Bronzeは週に1回のみ編集可能とし、変更案のメモを蓄積しながら編集判断を行う。

### ページ概要

**アクセス**
- サイドバーに「週次振り返り」メニューを追加
- メニュー順序（案）: ホーム → 記録 → カレンダー → 週次振り返り → 目標変遷 → 設定

**対象週**
- 直近7日間（月〜日）を表示
- 過去の週は閲覧のみ可（目標編集は不可）

---

### セクション1: 週次サマリー

```
今週のサマリー（2/10〜2/16）
─────────────────────────────
Gold達成率:      57%（先週比 +14%）
Gold時の満足度平均: 4.2（先週比 +0.3）
Bronze未達日:    1日（先週比 -2日）
```

**表示内容**
- 今週のGold達成率（達成日数 / 7日）と先週比
- Gold達成日の満足度平均と先週比
- Bronze未達日数と先週比
- 先週比は増減の方向が一目でわかるよう `+/-` で表示

---

### セクション2: Todo別の達成率分析

```
Todo別のGold達成への寄与分析
─────────────────────────────
[リファクタリング完了]  達成率 57%  ← 今週追加
[コードレビュー実施]    達成率 71%（先週比 +14%）
[設計書作成]           達成率 100%（先週比 ±0%）

仮にこのTodoをGoldから外すと:
  [リファクタリング完了] を外した場合 → 推定達成率 86%（+29%）
```

**表示内容**
- 今週のGold目標Todoそれぞれの個別達成率と先週比
- 今週新たに追加したTodoには「← 今週追加」ラベルを付与（先週比は表示しない）
- 「仮外し計算」: 各Todoを外した場合の推定Gold達成率と増減幅

**計算ロジック**
- 仮外し達成率 = 「対象Todoを除いた全TodoがチェックされているorそのTodoがあっても他が全部チェックされている日」/ 7日

---

### セクション3: Gold目標の編集

**編集権限**
- 制約なし。いつでも編集可能

**編集内容**
- 既存TodoのTodo内容を編集・削除可能
- 新規Todoを追加可能（上限なし）
- 削除はTodoを個別に削除できる（最低1件は残す）

**保存時の動作**
- 即時反映（その日以降の日報から新しいGold目標が表示される）
- 変更前の内容は `goal_level_history` に記録する（実装時に詳細設計）

---

### セクション4: Silver/Bronze目標の編集

**編集権限（Silver/Bronze）**
- 「月曜日以降に初めて週次振り返りページを開いた日」の当日中（23:59:59まで）のみ編集可能
- 編集可能な日 = その週の編集ウィンドウが開いた日
- 編集ウィンドウが開いたかどうかは `weekly_review_access_log` テーブルで管理
- 週の集計期間: 前週月曜〜前週日曜。月曜にページを開くと前週の集計が確定した状態で表示される

**編集内容（Silver/Bronze共通）**
- 既存TodoのTodo内容を編集・削除可能
- 新規Todoを追加可能
- 削除はTodoを個別に削除できる（最低1件は残す）
- 編集後、「変更理由」の入力が必須（自由記述、1文字以上）
- 変更理由は `goal_change_log` テーブルに記録

**編集不可時の表示**
- Silver/BronzeのTodo一覧は読み取り専用で表示
- 「月曜日以降に初めてこのページを開いた日に編集できます」と説明文を表示

---

### セクション5: 変更案メモ

**目的**
- 週の途中で「こうしたほうが良いかも」と思ったことをその場で書き留めておき、Silver/Bronze編集時に参照する

**仕様**
- 週次振り返りページ内に常時表示されるメモ欄
- 今週分のメモと、過去3週分のメモを一覧表示（過去分は読み取り専用）
- 今週分のみ編集・追記可能
- 1件ずつメモを追加する形式（リスト形式で時系列に並ぶ）

---

### データモデル変更（予定）

新テーブル `goal_change_memo`:

| カラム名 | 型 | NULL | 説明 |
|---------|-----|------|------|
| `id` | UUID | - | 主キー |
| `user_id` | UUID | - | 外部キー: auth.users |
| `week_start_date` | DATE | - | その週の月曜日の日付（週を一意に識別） |
| `content` | TEXT | - | メモ内容 |
| `created_at` | TIMESTAMPTZ | - | 作成日時 |

新テーブル `weekly_review_access_log`:

| カラム名 | 型 | NULL | 説明 |
|---------|-----|------|------|
| `id` | UUID | - | 主キー |
| `user_id` | UUID | - | 外部キー: auth.users |
| `week_start_date` | DATE | - | その週の月曜日の日付 |
| `edit_unlock_date` | DATE | - | 編集ウィンドウが開いた日付（月曜以降に初めて開いた日） |
| `created_at` | TIMESTAMPTZ | - | 作成日時 |

## 機能5: JSON一括出力

### 目的

蓄積したログをAIに渡し、「どういう行動をとった時に一日を満足して終えられているか」を分析するためのエクスポート機能。

### 出力内容

AI分析に必要な情報を1つのJSONファイルにまとめて出力する。

```json
[
  {
    "date": "2026-02-15",
    "achievement_level": "gold",
    "satisfaction": 4,
    "work_duration_minutes": 180,
    "journal_text": "今日は集中できた...",
    "todos": [
      {
        "type": "goal",
        "level": "gold",
        "content": "リファクタリングまで完了する",
        "is_achieved": true
      },
      {
        "type": "goal",
        "level": "silver",
        "content": "1機能を完成させる",
        "is_achieved": true
      },
      {
        "type": "other",
        "content": "散歩に行く",
        "is_achieved": true
      }
    ],
    "is_recovery": false
  }
]
```

**出力対象フィールド**
- `daily_records`: date, achievement_level, satisfaction, work_duration_minutes, journal_text
- `daily_todo_records`: todo内容（goal_todos/other_todosとJOINして内容を解決）、is_achieved
- Recovery達成有無

**出力対象外**
- ユーザー個人情報（user_id, email等）
- 提案バナー履歴
- タイムラインtodo（分析対象は日々の行動記録のみ）

**UI**
- 設定画面に「データをエクスポート」ボタンを追加
- ボタン押下でJSONファイルをダウンロード
- 出力期間の指定は初期版では不要（全期間出力）

## 優先順位と実装順序（案）

| 順序 | 機能 | 理由 |
|------|------|------|
| 1 | 作業時間計測 | データ収集は早いほど良い。ログが溜まるまで時間がかかる |
| 2 | 週次振り返りページ | Gold自由編集化を内包しており、即座に目標調整に使える |
| 3 | タイムラインページ | 新規ページ・新テーブルが必要で実装量が多い |
| 4 | JSON一括出力 | データが十分に溜まってから使う機能のため後回しで良い |
