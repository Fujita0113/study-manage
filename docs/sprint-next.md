## 概要

## 機能1: 作業時間計測

### 目的

日々の作業時間と達成レベル・満足度の相関を分析するための基礎データを収集する。

### 仕様概要

**計測開始**
- アプリ（ブラウザタブ）を開いた時点でタイマーをスタート

**計測継続/停止**
- タブがバックグラウンドになった時（`visibilitychange`イベント）: 一時停止
- タブが再度アクティブになった時: 再開
- タブを閉じた時（`beforeunload`イベント）: 停止 → 累積時間をlocalStorageに保存

**計測確定**
- 日報を保存した時点で、その日の累積時間を `daily_records` に保存して計測終了

**注意点**
- localStorageでセッション間の累積時間を引き継ぐ
- 当日の日報保存後に再びアプリを開いても、その日の計測は追加しない（確定済みのため）
- ブラウザクラッシュ等でlocalStorageに未保存の時間が残った場合は破棄（精度より堅牢性優先）

### データモデル変更（予定）

`daily_records` テーブルに以下を追加:

| カラム名 | 型 | NULL | 説明 |
|---------|-----|------|------|
| `work_duration_minutes` | INTEGER | 許容 | 作業時間（分）。日報保存時に確定 |

## 機能2: タイムラインページ（マイルーティン）

### 目的

「自分の理想の一日の流れ」をtodoリスト形式で育てていくページ。日付に依存せず蓄積し、毎日参照することで行動の改善を積み重ねる。

### 仕様概要

**ページの性質**
- 日付に紐づかない（マイルーティンとして永続的に育てていく）
- 既存の「その他todo」「Gold/Silver/Bronze目標todo」とは独立した管理

**表示形式**
- 時間帯タグ付きのリスト形式（例: `7:00 散歩に行く`、`9:00 開発作業`）
- 同一時間帯に複数のtodoを設定可能
- 時間帯順にソート

**todo操作**
- 追加: 時間帯（HH:MM形式）とtodo内容を入力して追加
- 削除（非表示）:
  - 削除時に「削除理由」の入力を必須とする
  - 削除されたtodoは物理削除せず、非表示フラグで管理（ソフトデリート）
  - 削除理由の選択肢（案）: 「満足度向上のため見直し」「今の状況でできない」「別のことを優先」「その他（自由記述）」
- 復活: 削除済みtodoを一覧から選択して復活させる機能

**日報との連携（変更済み）**
- 当初の「日報のtodoに自動反映しない」方針を変更。
- 旧来の「その他TODO」機能を完全に廃止し、代わりにこの「マイルーティン」を記録画面内に取り込み、達成チェックできるようにする。

### データモデル変更（予定）

新テーブル `timeline_todos` を追加:

| カラム名 | 型 | NULL | 説明 |
|---------|-----|------|------|
| `id` | UUID | - | 主キー |
| `user_id` | UUID | - | 外部キー: auth.users |
| `time_tag` | TEXT | - | 時間帯（例: "07:00"） |
| `content` | TEXT | - | todo内容 |
| `is_deleted` | BOOLEAN | - | ソフトデリートフラグ（default: false） |
| `delete_reason` | TEXT | 許容 | 削除理由（削除時に記録） |
| `created_at` | TIMESTAMPTZ | - | 作成日時 |
| `updated_at` | TIMESTAMPTZ | - | 更新日時 |

**アクセス**
- サイドバーのメニューに「ルーティン」として追加
- メニュー順序（案）: ホーム → 記録 → カレンダー → ルーティン → 目標変遷 → 設定

## 機能3: 週次振り返りページと日々の難易度メモ

### 目的

Gold/Silver/Bronze目標が、自分の生活リズムやその日の状況に対して適切な難易度に設定されているかを週1回見直し、チューニングするための機能。
「日々の記録時に感じた目標の難易度に関するメモ」を蓄積し、それを材料にして目標を修正する運用フローを実現する。

### 各目標の定義（再確認）

- **Bronze目標**: リカバリーモードのストッパー。午前中が完全にダメだったとしても、その後の頑張り次第で必ず達成できる最低限のライン。
- **Silver目標**: 午前中は頑張って動けたが午後力尽きた場合でも達成できているライン。
- **Gold目標**: 努力目標。状況に応じて頻繁に変更してよい。「Goldを達成した日は自ずと満足度が4以上になる」状態を担保するためのもの。

### 運用フローと機能要件

#### 1. 日報入力時の難易度メモ（日々の記録画面の拡張）

- **機能**: 日報（記録）画面において、その日の目標難易度（特にBronze/Silver）が適切だったかを振り返る入力欄を追加する。
- **入力内容**: 自由記述のテキスト（例：「講義が長引いてSilverがきつかった」「Bronzeはこれなら絶対できる」など）。任意入力でOK。
- **保存先**: `daily_records` テーブル、または新テーブルに保存する（実装時に詳細決定）。

#### 2. 振り返りのリマインド（ホーム画面）

- **機能**: 指定した曜日（例えば日曜日や月曜日など）になると、ホーム画面に「今週の振り返りをしましょう」という旨のダイアログやバナーを表示する。
- **アクション**: クリックすると週次振り返りページへ遷移する。

#### 3. 週次振り返りページ（チューニング画面）

- **アクセス**: サイドバー「週次振り返り」メニュー、またはホーム画面のダイアログから遷移。
- **対象**: 直近1週間のデータを中心に振り返る。

**セクション1: 今週の難易度メモ一覧**
- 1週間で蓄積された「日々の難易度メモ」を一覧表示する。
- ユーザーはこれを見ながら、今の自分のフェーズ（長期休み中、授業期間中など）にSilver/Bronze目標が合っているかを考える。

**セクション2: 目標のチューニング（編集）**
- **Silver/Bronze/Gold共通**: メモを参照しながら、画面上で直接Todoの内容を編集・削除・追加できる。
- 分析データではなく、ユーザー自身の感覚をもとに最適な難易度に調整することを主眼とする。

**セクション3: Gold目標の妥当性チェック（アラート機能）**
- 「Gold目標が努力目標として正しく機能しているか」をシステムがチェックし、該当する場合は指摘（アラート）を表示する。
  - **チェック例1**: 「明らかに達成不可能な目標がGoldに入っている」（例：何週間も連続で一度も達成されていないTodoがある等）。
  - **チェック例2**: 「Goldを達成しているのに、満足度が4未満の日がある」（＝Goldのハードルが低すぎる、または今の自分にとって本当にやりたい事とズレている可能性）。
- アラートを見て、ユーザーはGold目標の内容を見直す。

### データモデル変更（予定）

- 日報時の難易度メモを保存するカラム/テーブルの追加。
- 「週の振り返りを行ったステータス」を管理するテーブル（ホーム画面のダイアログ制御用）。
※ 詳細な設計は実装計画時に行う。

## 機能5: JSON一括出力

### 目的

蓄積したログをAIに渡し、「どういう行動をとった時に一日を満足して終えられているか」を分析するためのエクスポート機能。

### 出力内容

AI分析に必要な情報を1つのJSONファイルにまとめて出力する。

```json
[
  {
    "date": "2026-02-15",
    "achievement_level": "gold",
    "satisfaction": 4,
    "work_duration_minutes": 180,
    "journal_text": "今日は集中できた...",
    "todos": [
      {
        "type": "goal",
        "level": "gold",
        "content": "リファクタリングまで完了する",
        "is_achieved": true
      },
      {
        "type": "goal",
        "level": "silver",
        "content": "1機能を完成させる",
        "is_achieved": true
      },
      {
        "type": "other",
        "content": "散歩に行く",
        "is_achieved": true
      }
    ],
    "is_recovery": false
  }
]
```

**出力対象フィールド**
- `daily_records`: date, achievement_level, satisfaction, work_duration_minutes, journal_text
- `daily_todo_records`: todo内容（goal_todos/other_todosとJOINして内容を解決）、is_achieved
- Recovery達成有無

**出力対象外**
- ユーザー個人情報（user_id, email等）
- 提案バナー履歴
- タイムラインtodo（分析対象は日々の行動記録のみ）

**UI**
- 設定画面に「データをエクスポート」ボタンを追加
- ボタン押下でJSONファイルをダウンロード
- 出力期間の指定は初期版では不要（全期間出力）

## 優先順位と実装順序（案）

| 順序 | 機能 | 理由 |
|------|------|------|
| 1 | 作業時間計測 | データ収集は早いほど良い。ログが溜まるまで時間がかかる |
| 2 | 週次振り返りページ | Gold自由編集化を内包しており、即座に目標調整に使える |
| 3 | タイムラインページ | 新規ページ・新テーブルが必要で実装量が多い |
| 4 | JSON一括出力 | データが十分に溜まってから使う機能のため後回しで良い |
